Product architecture · v1.0

把一门课程，配置成一个独立智能体

学生端和教师端保持一套通用产品；课程设计者只配置角色、任务、活动、知识、AI边界、教师干预和数据规则。故宫六官作为第一份课程配置运行，不进入产品底层。

核心产品决策平台团队一次性开发通用底座；课程团队逐门定义内容；运行时依据配置组装体验。
平台底层AI对话、活动渲染、流程、协作、教师控制、AI工具和数据服务。
课程配置目标、阶段、N个角色、任务、知识、限制、评价、点位与媒体。
配置协议明确每项功能哪些参数由课程填写，以及如何影响双端运行。

01 / Product decision

不要做“每门课程一个小程序”，要做“同一个运行器加载不同课程”

当前故宫学生端和教师端已经证明了体验方向，但角色、页面、任务与判断逻辑仍然写死。继续按课程复制，会同时制造内容、产品、AI与数据四套债务。

核心问题

课程内容与产品逻辑耦合

“数龙官”“测坡官”和页面流程直接进入代码，新增角色就等于新增开发。

问题 02

AI只是播报层

AI没有拿到课程状态、学生证据、教师指令和知识出处，无法真正编排学习。

问题 03

数据只够展示进度

完成与否、位置、倒计时可以看见，但无法回答学生如何观察、推理、协作和修正。

目标形态：每门课程看起来拥有不同名称、角色、剧情、知识与AI人格，但底层始终运行同一套活动渲染、流程状态、智能体、教师控制与数据协议。

02 / Responsibility boundary

先分清两张清单：平台开发清单与课程设计清单

平台团队负责生产稳定能力，并为每项能力定义可配置参数。课程团队使用这些参数设计具体课程。课程上线时，运行时把两部分组装成学生看到的智能体体验。

PLATFORM · 一次开发，多课复用

产品与技术团队开发

这些能力写入学生端、教师端和服务端，课程团队无需重复制作。

AI主对话与课程状态每门课程必有；支持主动引导、等待、静默、总结和工具调用
学生端通用外壳小组、角色、当前任务、证据库、通知、离线与安全入口
教师端通用外壳全班态势、干预队列、小组详情、教学遥控和课后回看
活动组件与渲染器拍照、输入、录音、讨论、测量、标注、推演等通用界面
流程与协作引擎阶段、分支、并行、解锁、角色同步、同伴确认和教师覆盖
AI工具服务知识检索、图片理解、语音转写、语义分析、总结、网络检索和护栏

CONFIG CONTRACT
配置协议

03 / Platform architecture

五层架构：体验可以变，运行规则必须稳定

AI对话是学生看见的主入口，但它不能同时承担状态机、权限、数据校验和课程安全。关键流程必须由确定性系统控制，AI负责理解、生成、追问与适配。

04 / Course ontology

课程要先被拆成对象，才能被配置、运行和复用

建议统一使用下面的对象层级。角色数是数组长度，任务形式是活动组件组合，课程流程是一张阶段图，因此六个官、三个官或没有角色的课程都可以运行。

Course 课程→ Phase 阶段→ Role 角色→ Mission 总任务→ Task 子任务→ Activity 活动→ Artifact 学习产物
配置对象主要配置内容运行时如何使用解决的问题
Course课程名称、目标、适龄、场域、总时长、主题、版本创建一次课程会话并加载完整配置包区分不同课程与版本
Phase课前、集合、现场、推演、展示等阶段；进入条件与退出条件决定学生端当下可见内容与AI可透露范围控制主流程和教学节奏
RoleN个角色的人格、职责、能力梯度、人数、关系为成员生成不同任务视角和智能体上下文支持多角色与跨课程复用
Mission角色任务或小组总任务、核心问题、完成证据作为任务容器与解锁判断单元区分角色任务和全组任务
Task目标、步骤、地点、时间、前置、分支、失败策略流程引擎按状态选择下一步支持线性、并行与分支任务
Activity拍照、输入、讨论、定位等组件及其参数活动渲染器生成学生实际操作界面课程设计不再依赖开发页面
Artifact照片、观察记录、测量、证据卡、图谱、结论等进入小组证据库、评价与展示让任务结果成为可复用学习产物
Gate解锁条件、教师确认、同伴确认、时间与位置条件确定性判断下一阶段是否开放避免AI主观决定通关
Rubric完成标准、质量维度、观察点、评价主体AI反馈、教师评价和学生自评共同引用统一学习目标与数据解释

06 / Two mini-programs

双端共享同一课程状态，各自承担不同工作

学生端围绕“下一步学习行动”，教师端围绕“哪里需要干预”。两端都读取同一个课程会话、任务状态、证据与智能体运行记录。

学生小程序

对话是主导航，活动卡片是学习行为，证据库是长期上下文。

教师小程序

默认安静观察，异常与请求进入干预队列，教师动作可追踪。

开课控制台建班、分组、发布角色、设备与权限检查
全班态势阶段、位置、在线、任务与安全状态
干预收件箱学生呼叫、AI低置信、超时、偏离与争议
小组详情成员、对话摘要、证据、分歧与提示使用
教学遥控器广播、推送知识、加时、暂停、解锁、集合
课后回看作品、认知变化、协作与AI干预效果
阶段 01

AI说明当下目标

智能体依据阶段状态发起对话，并将课程目标转成学生可行动的问题。

阶段 02

动态出现活动卡

需要拍照时出现相机组件，需要讨论时进入小组房间；学生不必在功能菜单中寻找工具。

阶段 03

证据进入共享上下文

活动产物进入证据库，AI的下一次对话同时读取真实产物与聊天记录，并优先依据学习证据。

阶段 04

规则判断是否推进

系统检查配置的完成条件；缺少内容时由AI解释缺口，教师可人工覆盖。

阶段 01

开课前确认

教师确认分组、角色数量、路线、安全授权和课程版本，发布本次会话。

阶段 02

优先观察异常

系统汇总“需要关注”的小组：失联、超时、位置偏离、AI低置信或学生主动求助。

阶段 03

选择干预强度

教师可以只发一个追问，也可以加时、解锁、加入讨论或发布全班指令。

阶段 04

查看干预效果

系统记录干预前后小组是否恢复推进，为课程改版和教师复盘提供依据。

07 / Agent & knowledge

智能体由一组可配置课程资产共同构成

每门课程的“独立智能体感”来自课程人格、知识、角色视角、阶段策略和视觉表达；底层模型、检索、工具调用与安全规则由平台统一提供。

平台团队交付：AI对话编排器、课程知识检索、多模态工具、网络检索与来源标注、输出护栏和教师指令通道。
课程团队交付：下面八类课程资产，决定当前智能体在这门课里知道什么、引导什么、限制什么以及如何评价。
KB-01

知识类

事实、概念、史料、方法、地点与数据。每条知识都要带来源、可信等级、适用年龄和可见阶段。

KB-02

引导与限制类

本课程希望学生发现什么、AI应追问什么、哪些答案不能提前说、哪些价值判断必须保留开放性。

KB-03

流程策略类

每个阶段的对话目标、进入条件、活动调用、失败分支与结束条件。应结构化配置，不只写成自然语言。

KB-04

角色人格类

称呼、语气、身份、角色视角和表达难度。人格影响表达，不得改变事实、流程和权限。

KB-05

误区与诊断类

常见错误、容易混淆的概念、错误证据模式和相应追问，帮助AI识别学习困难并给出针对性引导。

KB-06

评价与示例类

评价量规、合格产物示例、反例、反馈句式和不同年龄表现，用于稳定AI反馈标准。

KB-07

场域与应急类

点位、路线、开放限制、替代任务、离线材料和安全要求，支持真实研学中的现场变化。

KB-08

教师策略类

什么情况只让AI继续，什么情况提醒教师，教师可发送哪些指令，以及指令对智能体行为的影响。

学生知识回答的来源优先级

1
课程知识库使用课程设计者审定内容，展示课程来源或知识卡编号。
默认
2
机构批准的扩展资料跨课程公共知识库、场馆资料和教师补充材料。
可追溯
3
模型通用认知只有课程允许时使用，明确提示“以下为AI通用知识，未在本课程资料中核验”。
需标注
4
网络检索必须获得课程策略允许；明确显示“根据网络信息”、来源链接、发布日期与检索时间。
强标注
5
无法可靠回答说明缺少可靠资料，可建议呼叫老师或记录为待研究问题，不编造答案。
安全降级
确定性上下文课程版本、阶段、角色、任务、权限、教师指令、已有证据
→
检索与策略组装检索可见知识，加载引导、限制、误区、评价与工具策略
→
结构化输出对话文本 + 活动调用 + 引用 + 状态建议 + 风险标记
不要把“主流程提示词”理解成一段超长 System Prompt。应拆成阶段策略、状态规则、知识权限、活动调用和输出协议。这样才能版本化、测试、审计和让教师临时干预。
08 / Teacher intervention

教师干预需要一套正式产品协议

干预包含触发原因、强度、作用范围、学生可见方式和解除条件。教师的指令进入智能体上下文，但不能悄悄篡改学生作品或伪装成AI自主判断。

L0静默观察AI正常引导，教师只看状态与摘要。正常推进
L1AI自动提醒提示遗漏、超时、证据不足或成员尚未确认。规则自动触发
L2教师远程轻推发送问题、知识卡、鼓励、提示或延长时间。教师选择小组
L3教师进入任务加入讨论、人工审核、解锁、退回、切换替代任务。学习或现场阻塞
L4安全接管暂停课程、紧急集合、逐人确认或关闭位置任务。安全优先
教师动作作用对象学生端表现系统记录
发送追问个人 / 小组 / 全班明确标注“教师提示”，进入当前对话教师、原因、内容、是否回应
推送知识卡角色 / 小组 / 全班作为课程资料出现，不冒充学生发现资料版本、阅读与引用情况
加时 / 暂停任务 / 小组 / 全班更新时间并说明教师操作原时间、变更值、原因
人工通过证据 / 任务 / 关卡显示教师确认，保留AI原判断覆盖对象、依据、操作者
切换替代任务点位任务 / 设备任务进入同目标的离线或室内版本触发原因、替代版本、学习目标等价性
紧急集合全班覆盖当前页面，要求确认并导航集合点广播时间、确认时间、未确认人员
09 / Learning data

数据同时记录操作结果与学习发生过程

数据项应由课程配置声明：采什么、为什么采、谁可见、保存多久、是否进入评价。平台提供统一事件协议，课程只选择需要的字段和评价映射。

会话与身份

  • 课程、版本、班级、场次
  • 小组、成员、角色分配
  • 设备、在线与权限

流程与行为

  • 阶段进入、任务开始与完成
  • 活动调用、停留、重试、跳过
  • 提示层级与求助行为

现场与安全

  • 到点、偏离与集合确认
  • 网络、离线队列、设备异常
  • 仅按必要频率采集位置

学习产物

  • 照片、视频、录音、输入、测量
  • 标注、证据卡、图谱与展示
  • 原始内容与AI整理内容分开保存

认知过程

  • 初始猜想、证据与结论
  • 确信程度、分歧与版本修改
  • 哪条新证据导致观点改变

协作过程

  • 角色领取、成员贡献
  • 讨论观点、投票与同伴确认
  • 共识形成和保留的少数意见

AI运行

  • 检索到的知识与来源
  • AI输出、活动调用、置信度
  • 护栏触发、降级和疑似误判

教师干预

  • 干预原因、级别与动作
  • 对象、时点与学生可见状态
  • 干预后是否恢复推进

评价与反思

  • 量规维度、评价主体与证据
  • 自评、互评、教师评价
  • 学习总结与后续问题

运行数据 ≠ 评价数据

位置、点击和用时用于保障课程运行，不应自动换算为学习能力。只有课程量规明确绑定的证据才能进入评价。

AI整理 ≠ 学生原作

必须保留学生原始表达、AI修改版本和接受记录，避免将AI生成内容误认为学生学习成果。

未成年人数据底线：默认最小化采集；明确监护与机构授权；位置和人像采用最短保存周期；教师只能查看所管理班级；网络检索、模型调用和第三方存储必须可审计。

这部分全部由课程设计团队负责。未来后台提供可视化编辑方式；当前先定义配置协议，也可以用文件、表格或脚本生产课程。

01. course.manifest｜课程总览
包含：课程ID、名称、版本、适龄、场域、总时长、学习目标、主题、发布状态。
作用：识别课程与版本，是所有对象的根。
02. phases.json｜阶段流程图
包含：阶段、进入/退出条件、顺序/并行关系、允许工具、答案披露等级、教师控制点。
作用：决定主流程，不依赖AI自由发挥。
03. roles.json｜N个角色
包含：角色名称、人数、职责、人格、能力梯度、任务关系、共享与私有信息。
作用：角色数量和任务视角完全数据化。
04. missions.json｜角色与小组任务
包含：核心问题、任务步骤、地点、活动序列、产物、完成条件、替代任务。
作用：把课程教学设计转为可运行任务。
05. activities.json｜活动组件参数
包含：组件类型、字段、素材、校验、AI分析、教师权限、数据采集。
作用：同一拍照组件在不同任务中呈现不同要求。
06. knowledge/｜知识资产
包含：知识单元、来源、可信度、标签、可见阶段、适用角色、年龄与引用格式。
作用：成为智能体的首要回答依据。
07. agent.policy｜智能体策略
包含：人格、阶段目标、引导方法、禁答内容、误区、提示层级、网络策略、输出协议。
作用：形成课程独立智能体体验，同时可测试和审计。
08. teacher.policy｜教师干预策略
包含：异常规则、告警等级、可执行动作、作用范围、人工覆盖与安全流程。
作用：把教师端能力和课程节奏绑定。
09. rubric.json｜评价量规
包含：维度、等级、观察点、关联产物、评价主体与AI可用范围。
作用：指导反馈与课后评价，避免只统计完成率。
10. data.schema｜数据字典
包含：采集字段、用途、可见角色、保存周期、是否进入评价、脱敏规则。
作用：课程上线前完成数据治理。
11. theme.json + assets/｜视觉与媒体
包含：课程颜色、智能体名称、角色徽标、音视频、背景、知识卡模板。
作用：让不同课程拥有独立视觉，但不改变产品结构。

11 / Forbidden City mapping

故宫课程作为第一份标准课程配置运行

角色数组填入六个官，现场任务分别组合不同活动；每个小组完成六个角色任务后，共享产物进入同一个总任务。下一门课程只替换配置对象。

课程阶段图

PHASE 01沉浸影片教师统一触发；学生记录初始猜想
PHASE 02组队领卡扫码激活六种身份；建立小组会话
PHASE 03故宫采证六角色并行任务；AI现场陪伴
PHASE 04小组推演证据图谱；形成“不易积水”解释
PHASE 05展示与璇玑分享、讲解、影片与认知修正
角色配置核心活动组合关键产物AI策略
数龙官拍照 → 标注 → 计数/输入 → AI追问螭首结构与局部排水证据观察教练，不给出总数量答案
测坡官定位 → 多次测量 → 小组讨论 → 提交判断坡向数据与可靠性说明方法教练，提醒重复测量
寻沟官拍照 → 分类 → 位置记录 → 关系标注明沟、钱眼与暗沟推测证据教练，区分观察与推断
引河官路线采集 → 视频/照片 → 地图标注 → 讨论内金水河走向与功能假设系统教练，追问弯曲原因
护城官定位 → 观察记录 → 测量/估算 → 同伴确认护城河调蓄与外围关系尺度教练，避免无证据夸大
真相官证据审查 → 来源分类 → 反例问题 → 小组讨论证据等级与未解问题质疑教练，不提前透露璇玑反转
小组总任务证据库 → 推演画布 → 共识确认 → 展示水系图、原因模型、证据链、保留问题推理伙伴，只指出断点与矛盾
最终任务解锁 = 六张身份已激活 + 六类必要证据已提交 + 每类至少一名同伴确认。AI质量判断只提供提示，不直接作为唯一通关分数。
12 / Course production

未来设计一门新课程，需要完成哪些内容

课程生产从学习证据倒推任务，再编写智能体对话与引导策略。完成下列八步后，平台即可生成对应的学生端、教师端和课程智能体。

定义学习目标与最终产物

明确学生最终要能够解释、制作、比较或迁移什么，以及用什么真实产物证明。

设计阶段图

确定集体环节、角色任务、总任务、展示与反思的先后、并行和解锁关系。

定义N个角色

说明每个角色为什么存在、掌握什么独特证据、如何与其他角色形成信息互赖。

组合任务活动

从组件库选择拍照、输入、讨论、测量、定位等活动，配置产物与完成条件。

建设课程知识资产

整理事实、史料、方法、误区、出处、阶段权限和网络检索策略。

配置AI教学策略

定义各阶段AI要推动的思考、不能透露的内容、提示层级与多模态分析标准。

配置教师与安全策略

定义教师告警、可执行动作、人工覆盖、替代任务、位置与紧急流程。

定义数据并进行模拟课

声明数据用途与保存规则，用虚拟小组跑通正常、失败、离线、误判和教师接管路径后发布。

知识库来源、权限与可信度
智能体策略引导、限制、误区与工具
教师策略告警、干预与安全
评价量规证据、维度与主体
数据字典用途、权限与保存期

13 / Build roadmap

先验证“配置能生成课程”，再建设完整后台

第一阶段不必急着做一个庞大的课程编辑后台。先用故宫课程把配置协议、双端运行和智能体边界跑通，后台只是后续提高生产效率的工具。

MVP / 优先

课程运行底座

  • 课程、阶段、N角色、任务与活动配置
  • 学生AI主对话 + 动态活动卡
  • 基础图片理解、语音转写和文本语义分析
  • 小组状态、证据库与总任务
  • 教师态势、呼叫、加时、推送、解锁与集合
  • 课程知识RAG、阶段禁答和来源标注
  • 统一学习事件与审计日志
PHASE 2

增强学习能力

  • 视频、复杂图谱与跨媒体联合分析
  • 讨论主持、共识与推演画布
  • 不同年龄的自适应提示
  • 离线现场任务与同步冲突处理
  • 课后证据与认知变化报告
PHASE 3

课程生产后台

  • 可视化阶段图编辑器
  • 活动组件配置器
  • 知识与限制管理
  • 智能体策略测试台
  • 课程版本、灰度与复用市场

建议下一步直接做什么

先建立一份正式的“课程配置字段清单”，再用故宫课程逐字段填写并验证。随后选择第二门结构明显不同的课程进行盲测：如果不改底层代码也能生成可用的双端流程，这套抽象才算成立。
