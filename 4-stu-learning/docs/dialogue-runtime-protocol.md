# 研学智能体平台·学生对话运行协议

> 版本：v1.1 运行基线（2026-07-22）  
> 适用范围：学生端小程序、学生端 H5、智能体服务端、课程编译器、教师干预通道  
> 性质：平台级强约束，所有课程共用  
> 关联文档：`PLAN.md`、`docs/frontend-architecture.md`、`6-lessons/COURSE-SUBMISSION-SPEC.md`

---

## 0. 核心决策

学生端由对话驱动学习流程。每一轮对话都要先理解学生当前的表达，再决定回应聊天、处理情绪、解答问题、引导当前小步、调用工具、等待或请求教师介入。

本协议做出以下七项架构决策：

1. 课程学习状态和自然对话状态分开管理。
2. 平台记录“正在等学生回答什么”，短回答优先结合该状态理解。
3. 每轮只执行一个主要对话动作，最多提出一个明确问题。
4. 大模型输出结构化的回合决策，平台校验后才更新学习状态和调用工具。
5. 课程进度依据位置事件、工具结果、证据评估或教师指令推进；仅 `user_confirm` 小步允许学生用明确完成表达推进，自由文本不得越过其他课程验收条件。
6. 课程内容始终来自课程配置。对话意图、情绪处理、打断恢复、安全、学段适配和工具调度属于平台底层。
7. 对话界面中一次只突出一个主要行动；阶段卡、对话气泡和工具卡不重复同一段内容。

## 1. 范围与目标

### 1.1 本协议解决的问题

- 学生输入“好了”、“嗯”、“行”时，能结合上一问判断所回答的内容。
- 学生寒暄、跑题、抱怨、拒绝、疲惫或输入无语义字符时，对话可以自然处理并保留学习进度。
- 学生知道大致任务数量，絮絮依据课程配置每次只带领一个小步。
- 课程可拥有 N 个角色、N 个角色阶段和不同工具，底层对话逻辑保持一致。
- 小学、初中、高中学生使用同一运行引擎，信息量、问题开放度、等待时间和脚手架深度随学段调整。
- 位置、时间、小组状态和教师指令能参与回合决策。

### 1.2 不在本协议中定义的内容

- 某一门课的具体知识、答案、史料、角色、地点和学生任务。
- 某一个工具的界面视觉细节。
- 某一门课的教师教学策略和阈值。

## 2. 统一术语

| 术语 | 运行时字段 | 定义 | 故宫课程示例 |
|---|---|---|---|
| 课程阶段 | `coursePhase` | 全体学生共同经历的大流程 | 沉浸导入、现场采证、小组推演 |
| 角色阶段 | `roleStage` | 当前角色需要完成的一个主任务 | 数龙官的观其形、算其数、验其差 |
| 小步 | `step` | 角色阶段内一次可理解、可执行、可验证的行动 | 选一处螭首并拍全景 |
| 对话回合 | `turn` | 学生一次输入到智能体一次完整响应 | 学生说“我到了”到絮絮回复 |
| 待回答问题 | `pendingQuestion` | 絮絮上一轮明确等待学生回答的唯一问题 | 是否已到任务点 |
| 对话动作 | `dialogueMove` | 本轮智能体要完成的主要交互目的 | 确认到达、承接情绪、给出提示 |
| 工具调用 | `toolRequest` | 由平台校验后发送给客户端的能力请求 | 打开相机、导航、呼叫教师 |
| 证据 | `evidence` | 学生通过工具或表达提交的可评估学习产出 | 照片、估算值、语音、示意图 |

UI、埋点、API 和课程编译器应使用这组术语，避免继续将课程阶段、角色任务和小步统称为“阶段”。

## 3. 平台层与课程层职责

### 3.1 平台必须开发

| 能力 | 作用 |
|---|---|
| 回合理解器 | 结合待回答问题、历史和环境判断学生意图 |
| 对话策略器 | 按优先级选择本轮对话动作 |
| 对话状态管理 | 管理待回答问题、已确认信息、情绪、误解和重复 |
| 学习状态机 | 管理课程阶段、角色阶段、小步、工具、证据和转移 |
| 学段适配器 | 统一控制字数、句长、问题开放度、等待时间和主动提示频率 |
| 脚手架引擎 | 结合课程分级提示与学生实时表现调整引导深度 |
| 检索和来源策略 | 按当前任务取最小必要知识，管理课程、AI 已有知识和网络来源 |
| 工具调度和校验 | 控制哪些工具在当前状态可用，校验参数并处理结果 |
| 安全与教师优先级 | 保证安全事件和教师广播可以打断任何对话状态 |
| 弱网与失败恢复 | 保留状态，处理重试、离线缓存和可恢复的交互 |
| 重复检测 | 阻止连续发送语义相同的提问或流程播报 |
| 对话渲染协议 | 把回复、阶段卡、工具卡、快捷回复和教师消息组合为稳定 UI 事件 |

### 3.2 课程团队必须配置

| 内容 | 用途 |
|---|---|
| 课程阶段 | 定义全体学习流程、时长、地点、教师触发和结束条件 |
| 角色和角色阶段 | 定义 N 个角色各自的大任务及顺序 |
| 小步 | 定义一次希望学生执行的行动、产出和完成方式 |
| 工具参数 | 决定拍照数量、表单字段、画布底图、答题内容和仿真规则 |
| B1 领域知识 | 提供可回答的知识、来源、适用角色和揭示时机 |
| B2 引导规则 | 定义该引导向哪个教学目标发展，包括典型触发与追问策略 |
| B3 课程限制 | 定义防剧透、跨角色隔离、揭示时机和禁止行为 |
| B4 流程提示 | 定义每个课程阶段和角色的教学方向 |
| B5 评估标准 | 定义小步验证、角色阶段通过和能力评估 |
| B6 分级脚手架 | 提供不同难度的提示内容和课程特定触发阈值 |
| 常见误区 | 帮助 AI 识别学生当前理解并选择反馈 |
| 转场与总结 | 角色阶段通过后告诉学生发现了什么、下一阶段前往何处 |

## 5. 对话生命周期

### 5.1 标准状态

```text
WAIT_FOR_TEACHER_RELEASE
  → ORIENT_ROLE
  → CONFIRM_ARRIVAL
  → CONFIRM_READINESS
  → INTRODUCE_ROLE_STAGE
  → GUIDE_CURRENT_STEP
  → WAIT_FOR_STUDENT
  → WAIT_FOR_TOOL_RESULT / EVALUATE_RESPONSE
  → GIVE_FEEDBACK
  → GUIDE_CURRENT_STEP / REFLECT_ROLE_STAGE
  → TRANSITION_ROLE_STAGE
  → ROLE_COMPLETE
```

任何时刻都可以进入：

```text
PAUSED_BY_TEACHER
SAFETY_ESCALATION
EMOTIONAL_SUPPORT
CONVERSATION_REPAIR
NETWORK_RECOVERY
```

### 5.2 入场确认

入场时一次只确认一项信息：

1. 如果当前角色阶段需要位置，先确认到达。
2. 位置围栏已验证时，可跳过口头到达确认。
3. 到达确认后，单独确认是否准备开始。
4. 学生一句明确表达同时包含“已到达 + 已准备”时，两个信息可以同时更新。
5. 只有所需条件都满足后，才进入角色阶段介绍。

### 5.3 角色阶段介绍

进入新角色阶段时，客户端显示一张 `roleStageCard`，内容只包含：

- 当前是第几个角色阶段；
- 角色阶段名称；
- 主要学习目标；
- 预计时长，读取角色阶段的 `建议时长`；缺省值为 15 分钟。

絮絮的同轮对话只补充当前第一个行动，不复述卡片内容。

### 5.4 小步学习循环

```text
ORIENT 定向
  → ACT 行动
  → OBSERVE 观察
  → EXPRESS 表达
  → EVIDENCE 提交证据
  → FEEDBACK 反馈
  → REFLECT 反思
  → TRANSITION 转移
```

课程可以按小步实际需要省略其中部分环节。平台每次只公开当前小步，不提前展开后续小步。

## 6. 每轮决策管线

### 6.1 输入

回合决策器每次只获取必要上下文：

- 学生当前输入；
- 最近 8–12 条对话或经压缩的对话摘要；
- `pendingQuestion` 和已确认信息；
- 当前 `coursePhase / roleStage / step`；
- 当前角色阶段装配的引导与脚手架、当前 Step 精确引用的限制；AI 验收回合再加入评估引用标签和 B5 原文；
- 位置、停留时间、阶段时间和小组状态；
- 最新教师指令；
- 当前可用工具和工具状态。

不将整门课的 B1–B6 在每轮完整携带。

### 6.2 处理步骤

```text
1. 硬事件检查
2. 回合理解
3. 待回答问题解析
4. 学生状态评估
5. 对话动作选择
6. 课程内容最小检索
7. 生成结构化回合决策
8. 安全、剧透、工具和状态校验
9. 流式发送学生可见文本
10. 执行工具或 UI 事件
11. 写回状态、消息和埋点
```

### 6.3 模型输出协议

规则层无法可靠理解、且当前存在 `pendingQuestion` 的自由文本回合，使用一次结构化模型调用。已确定的寒暄、情绪、抱怨、安全、位置、快捷回复和工具结果走快速路径。

```json
{
  "speechAct": "affirm",
  "answersPendingQuestion": true,
  "pendingAnswer": "yes",
  "emotion": "neutral",
  "taskRelation": "current_flow",
  "confidence": 0.96,
  "dialogueMove": "ask_readiness",
  "reply": "好，位置确认了。你想现在开始，还是先等一下队友？"
}
```

结构化理解结果只能表达语义判断，不能直接修改学习状态。平台根据当前 `pendingQuestion.kind`、`pendingAnswer` 和置信度更新对应槽位；模型无权自选下一个问题、完成小步或宣布工具成功。普通自由对话可以流式输出，并通过实时限制过滤。

### 6.4 快速路径

以下事件的语义已经确定，可以绕过自由文本意图理解：

- 学生点击快捷回复；
- 工具结果回传；
- 地理围栏事件；
- 教师指令；
- 页面可见性和状态心跳；
- 安全按钮和呼叫教师按钮。

简单寒暄可使用平台快速回复池，但不更改任何学习状态。

## 7. 优先级、打断与恢复

### 7.1 回合优先级

```text
P0 人身安全 / 紧急教师广播
P1 学生明确求助 / 教师控制指令
P2 学生情绪、抱怨、拒绝和对话修复
P3 学生正在询问的问题
P4 待回答问题解析
P5 当前小步引导、证据反馈和任务转移
P6 主动提醒
```

### 7.2 打断规则

- 安全事件立即中止正在生成的普通回复。
- 教师暂停后，不再打开新任务工具或推进角色阶段。
- 学生表达明显不满时，本轮进入 `CONVERSATION_REPAIR`，暂停课程催促。
- 学生寒暄、感谢、告别、表达情绪、提出课程问题、社交闲聊或请求老师时，挂起当前待回答问题，保留学习状态并先回应当前表达。
- 打断后收到“好”等模糊肯定不确认旧问题；学生明确说“继续”时，平台依据 `confirmedSlots` 重新计算下一项入场问题或当前小步。
- 恢复时不重播完整任务，只提醒当前唯一待办。

### 7.3 对话修复规则

当学生说“你怎么一直重复”、“你有毒吧”或同类表达时，回复应完成：

1. 承认具体问题；
2. 停止当前重复策略；
3. 保留已确认状态；
4. 挂起当前未回答问题，避免学生下一句“好”误确认旧问题；学生明确说“继续”后依据已确认状态重新计算待办。

示例：

> 你说得对，刚才的回应让人很烦。我先停下来听你说；想继续时告诉我“继续”就行。

## 8. 学生表达分类

| 类型 | 示例 | 平台动作 | 是否推进学习状态 |
|---|---|---|---|
| 寒暄 | 你好、早上好 | 自然回应 | 否 |
| 肯定回答 | 好了、嗯、行、可以 | 优先解析当前 `pendingQuestion` | 仅更新可被该回答确认的状态 |
| 否定回答 | 还没、不行、等一下 | 保留进度、提供等待或帮助 | 否 |
| 模糊回答 | 差不多、应该吧 | 只澄清当前未决信息 | 否 |
| 无语义输入 | ==、……、单个符号 | 一次轻量澄清 + 快捷选项 | 否 |
| 情绪 | 好烦、好累、害怕 | 承接情绪、暂停催促 | 否 |
| 抱怨 | 你有毒吧、怎么一直重复 | 进入对话修复 | 否 |
| 拒绝 | 我不做了 | 了解原因，可降级、休息或求助 | 否 |
| 位置问题 | 太和殿在哪、我迷路了 | 位置答复、导航或安全升级 | 仅位置验证事件可推进 |
| 操作问题 | 照片怎么传 | 直接给出操作说明 | 否 |
| 当前小步求助 | 我不知道拍什么 | 调用当前小步脚手架 | 否 |
| 学科知识问题 | 螭首为什么能排水 | 检索允许公开的课程知识 | 否 |
| 索要答案 | 直接告诉我有多少只 | 检查揭示时机，给引导或允许答案 | 否 |
| 表达发现 | 我看到下层更多 | 记录发现，选择追问、证据或反馈 | 按小步条件判断 |
| 宣称完成 | 我做完了 | 检查完成方式和工具结果 | 仅 `user_confirm` 类小步可直接推进 |
| 提交证据 | 照片、文字、表单结果 | 执行 B5 评估或请教师处理 | 评估通过后推进 |
| 小组问题 | 队友还没到 | 读取小组状态，等待或通知教师 | 按课程协作条件判断 |
| 安全求助 | 我受伤了、找不到队伍 | 停止课程并呼叫教师 | 否 |

## 9. 平台对话动作库

`dialogueMove` 使用稳定枚举，课程不得自行增加平台动作类型。

| 类别 | 动作 |
|---|---|
| 社交 | `respond_greeting`、`respond_social`、`respond_meta` |
| 状态确认 | `ask_arrival`、`confirm_arrival`、`ask_readiness`、`confirm_readiness`、`clarify_one_slot` |
| 情绪和修复 | `acknowledge_emotion`、`offer_pause`、`repair_conversation`、`handle_refusal` |
| 教学定向 | `introduce_role_stage`、`orient_step`、`resume_step`、`summarize_role_stage` |
| 引导 | `ask_observation`、`ask_reason`、`ask_compare`、`give_scaffold`、`surface_misconception` |
| 知识 | `answer_course_knowledge`、`answer_model_knowledge`、`answer_network_knowledge`、`protect_reveal` |
| 证据 | `request_evidence`、`acknowledge_evidence`、`give_evidence_feedback`、`request_retry` |
| 工具 | `open_task_tool`、`open_navigation`、`open_group_tool`、`call_teacher` |
| 等待和提醒 | `wait_silently`、`check_in`、`nudge_current_step` |
| 安全 | `stop_for_safety`、`escalate_teacher`、`broadcast_teacher_message` |

每轮只有一个 `primaryDialogueMove`。工具请求和 UI 事件可以作为该动作的支持结果。

## 10. 待回答问题协议

### 10.1 核心规则

- 任何时刻最多存在一个活动 `pendingQuestion`。
- 新问题发送前，平台要结束、替换或挂起旧问题。
- “好了”、“嗯”、“行”等短回答优先对应当前待回答问题。
- 已确认的信息不再询问。
- 模糊回答只澄清当前一个未决信息。
- 一次不向学生提出两个必须分别回答的问题。
- 连续两次无法理解时，停止开放文本澄清，改用 2–3 个快捷选项。

### 10.2 快捷回复默认模板

| 问题 | 快捷回复 |
|---|---|
| 到达确认 | `已到达` / `还在路上` / `需要导航` |
| 准备确认 | `现在开始` / `等一下` |
| 模糊表达 | 依据当前问题生成两个互斥选项 |
| 继续还是求助 | `继续试试` / `给我一个提示` / `呼叫老师` |

## 11. 小步完成和证据协议

### 11.1 小步完成方式

| `completionMode` | 条件 | 适用场景 |
|---|---|---|
| `user_confirm` | 学生明确确认完成 | 阅读、观看、低风险现场观察 |
| `tool_result` | 指定工具返回有效结果 | 拍照、录音、表单、画板、扫码 |
| `ai_evaluation` | 证据符合 B5 评估标准 | 图片识别、开放表达、示意图评估 |
| `teacher_confirm` | 教师审核通过 | 高风险、开放式成果或集体展示 |
| `location_event` | 位置、精度和停留时间满足条件 | 到达特定观察点 |
| `compound` | 多个条件按 AND/OR 组合 | 位置 + 照片、小组全员 + 教师确认 |

`ai_evaluation` 的运行顺序固定为：平台先执行必填、字数、张数、时长等确定性校验；再按 `知识引用` 检索当前可揭示知识，按 `限制引用` 精确解析当前 Step 的最小限制片段，把 Step 目标、证据要求、常见误区、评估引用标签、B5 原文、限制片段、结构化工具值和最多两张压缩图像交给评阅模型。模型只返回 `passed`、一句反馈和缺失项。`passed=true` 才记录 Step 完成；`passed=false` 保留草稿并停留原 Step。达到 `最大尝试` 后提示学生呼叫老师，后续可由教师端正式审核队列接管。

视觉评阅只接收当前 Step 的图片、实物识别照片或画板结果。浏览器临时 URL、录音器、媒体流和私有答案不会进入评阅请求。评阅反馈再次经过课程限制检查，命中未解锁内容时改为通用证据提示。

### 11.2 分层推进条件

- **小步完成**：当前 `completionMode` 满足。
- **工具完成**：客户端回传的工具结果通过参数校验。
- **角色阶段通过**：所有必做小步完成，且 B5 验收或教师条件满足。
- **课程阶段推进**：全局 FSM 中的角色、小组、时间和教师条件满足。

学生口头说“做完了”时，平台先检查当前小步的 `completionMode`。只有 `user_confirm` 可直接推进。

## 12. 课程小步配置契约

现有 `引导步骤：小步1；小步2；小步3` 可继续作为演示和兼容格式。正式课程使用以下结构化契约。

```md
### 角色阶段1：定坐标

- id：map-stage-1
- 阶段：phase-2
- 配置：辨认展陈地图的方向、图例和来源，再开始空间判断
- 通过条件：必做小步全部完成并提交可复核证据
- 推进方式：auto_after_validation
- 位置模式：point
- 地点：长征路线地图与模型展区
- 到达验证：manual
- 建议时长：15min
- 无操作提醒：3min
- 提醒冷却：2min
- 最大主动提醒：2

#### Step 1：确认展项

- id：map-locate-exhibit
- 小步目标：确认当前地图的展项名称和空间范围
- 学生行动：拍一张同时包含展项名称和地图全景的照片
- 位置：inherit
- 完成方式：ai_evaluation
- 功能模块：A01(拍照)
- 工具参数：{"photo":{"prompt":"照片需同时看清展项名称和地图全景","minCount":1,"maxCount":2,"accept":"image/*"}}
- 证据要求：至少1张照片，能够识别展项名称与地图整体范围
- 知识引用：K-10, K-11
- 引导引用：guidance/map-strategist.md#任务1
- 限制引用：restrictions.md#核心路线限制
- 评估引用：evaluation.md#S1, evaluation.md#S3
- 脚手架引用：scaffolds/map-strategist.md#任务1
- 常见误区：只拍地图局部，无法判断来源和整体方向
- 最大尝试：3
- 失败处理：提示调整角度并保留待核字段
- 教师介入：展项不可拍摄或无法找到指定区域
- 通过后：step:map-capture-source
```

### 12.1 小步必填字段

| 字段 | 作用 |
|---|---|
| `id` | 课程内稳定唯一 ID |
| `小步目标` | 该步骤要形成的学习结果 |
| `学生行动` | 智能体当前告诉学生的唯一主要行动 |
| `位置` | `inherit`、位置对象或 `none`；任何小步都必须显式存在该字段 |
| `完成方式` | 平台用哪种方式判定小步完成 |
| `证据要求` | 完成方式为工具或评估时的验收内容 |
| `通过后` | 下一小步或角色阶段转移 |

### 12.2 按需字段

- 功能模块和工具参数；
- 知识 ID，以及引导、限制、评估和脚手架文件引用；
- 预期回答类型；
- 常见误区；
- 最大尝试、失败处理和教师介入阈值；
- 是否可跳过、是否需要教师确认；
- 按学段区分的行动文案或特殊适配。

### 12.3 位置字段约束

每个角色阶段必须提交 `location`，每个小步也必须提交 `location`。课程没有位置要求时仍应显式填写，防止平台依靠任务文案猜测地点。

```json
{
  "location": {
    "mode": "none",
    "placeId": null,
    "placeName": null,
    "coordinate": null,
    "radiusMeters": null,
    "minimumDwellSeconds": null,
    "arrivalVerification": "none"
  }
}
```

位置模式统一为：

- `none`：无需地点，其他值允许为 `null`；
- `inherit`：小步继承所属角色阶段的位置；
- `point`：用于展示地点和导航，不自动判定到达；
- `geofence`：通过坐标、精度、围栏半径和停留时间判定；
- `route`：任务发生在一条线路上；
- `area`：任务发生在一个多边形区域内。

课程 Markdown 可以使用中文字段提交，课程编译器统一转换为以上对象。地点名称、坐标、围栏、停留时间和到达验证方式应随对应位置模式提交。

## 13. 多学段适配协议

课程 B6 提供教学内容的分级提示，平台学段适配器统一控制对话形态。

| 学段 | 单轮建议字数 | 一次行动 | 问题方式 | 默认引导强度 |
|---|---:|---|---|---|
| 小学低年级 | 15–30 | 极具体，可用图片和语音 | 指认、二选一、简短描述 | 主动给出第一层脚手架 |
| 小学高年级 | 30–50 | 具体行动 + 一个观察点 | 观察、比较、简单原因 | 短时等待后提示 |
| 初中 | 50–80 | 可包含方法选择 | 半开放、证据解释、估算 | 鼓励先尝试，卡住后提示 |
| 高中 | 80–120 | 允许自主规划 | 开放、系统、反事实、证据质疑 | 降低主动提示频率 |

字数是默认目标，安全说明、工具故障和必要的知识解释可以超出。

### 13.1 统一表达标准

- 一轮最多一个主要问题。
- 一次只要求学生完成一件事。
- 先承接学生当前表达，任务关联只在需要时出现。
- 操作、安全、导航和允许公开的事实可直接说明。
- 发现型任务答案依据课程限制使用追问或脚手架。
- 不对学生的人格、智力和品德作评价。
- 反馈要指向具体观察、方法或证据。

## 14. 知识检索、限制与来源

### 14.1 最小检索策略

系统默认携带：

- 平台安全、隐私和教师边界；
- 当前对话状态；
- 当前课程阶段、角色阶段和小步；
- 当前角色阶段按顺序装配的 B2 引导与 B6 脚手架；
- 当前 Step 通过 `限制引用` 精确解析的 B3 最小片段；
- `ai_evaluation` 时使用评估引用标签和完整 B5 文件；按标题裁剪单条 B5 量规仍待开发。

学生提出知识问题时，再按以下顺序检索：

1. B1 课程知识；
2. AI 已有认知；
3. 网络信息（课程和教师允许时）。

### 14.2 来源显示

| 回复类型 | 学生端来源标签 |
|---|---|
| 寒暄、情绪、流程、工具操作、导航 | 不显示 |
| 课程知识 | `课程资料 · 来源名称` |
| AI 已有认知 | `根据AI已有知识` |
| 网络检索 | `根据网络信息 · 来源链接` |

普通流程气泡不显示 `[课程配置｜角色与流程]`。

### 14.3 限制执行

回复发送前要经过：

- 安全边界；
- 未成年人话题边界；
- 课程防剧透与揭示时机；
- 跨角色知识隔离；
- 工具权限和参数校验；
- 证据分析可用性；
- 来源标注完整性；
- 重复和信息过载检查。

## 15. 工具调用协议

### 15.1 调用原则

- 只调用当前 `step` 配置或平台安全能力允许的工具。
- 模型提出 `toolProposal`，工具调度器检查后生成真实调用。
- 模型不能声称工具成功、位置已验证或任务已通过。
- 工具结果使用 `toolCallId`、`stepId` 和会话 ID 校验，过期结果不更新学习状态。
- 相同工具在等待结果期间不重复打开。

### 15.2 工具失败

- 保留学生草稿、已选照片和当前小步。
- 说明具体失败原因，如照片数量不足、上传失败或定位不可用。
- 提供重试、保存后稍后提交或呼叫教师。
- 工具失败不得转译为“AI 没有看懂”或伪造证据反馈。

## 16. 教师干预协议

平台必须提供以下教师动作：

- 开启或锁定角色领取；
- 开始、暂停、恢复或结束课程阶段；
- 定向或全体发送消息；
- 加时、减时或调整主动提醒阈值；
- 确认到达、验收学生证据、退回重做；
- 调高或调低某位学生的脚手架等级；
- 允许跳过可选小步或强制转移；
- 查看学生当前位置、状态、待回答问题和最近对话摘要；
- 处理学生呼叫和安全预警。

课程可以配置教师介入阈值，教师能力类型和权限属于平台固定能力。

## 17. 客户端渲染协议

### 17.1 事件类型

当前 SSE 传输协议使用下列稳定事件：

| 事件 | 作用 |
|---|---|
| `assistant.delta` | 模型首批文字与后续流式分片 |
| `assistant.completed` | 一条智能体消息的最终文本、来源和意图 |
| `stage.started` | 角色阶段卡：序号、名称、主任务、地点、小步数量和 `suggestedSeconds` |
| `ui.quick_replies` | 与唯一 `pendingQuestion.id` 绑定的快捷回复 |
| `tool.requested` | 打开导航、课程活动或呼叫老师 |
| `state.updated` | 写回学习、对话、位置和当前小步状态 |
| `agent.error` | 校验错误或真实连接错误；两类错误必须区分 |

`tool_result` 目前作为学生端到服务端的输入 `type` 传输；证据反馈通过 `assistant.completed` 和 `state.updated` 返回。后续若拆成单独事件，要同时升级客户端和协议版本。

### 17.2 单回合组合限制

一个普通回合最多包含：

```text
1 条智能体气泡
+ 1 张必要的信息或工具卡
+ 0–3 个快捷回复
```

阶段启动属于流程转场例外，可依次出现“阶段卡 → 当前第一个行动气泡 → 任务工具卡”，三项间隔 500ms，且内容各司其职。安全警报和教师广播使用独立高优先级渲染。

### 17.3 内容分工

- `roleStageCard` 显示角色阶段名称、目标、进度和时长。
- 智能体气泡只告诉学生当前一个行动或回应学生刚才的问题。
- 工具卡只呈现当前小步需要的操作。
- 快捷回复只在问题答案集较小或连续误解时出现。

### 17.4 消息节奏

- 页面入场后可以稍后显示第一条消息，建议 300–600ms。
- 只有语义独立的多条消息才使用 450–700ms 间隔。
- 可以用一条气泡清楚表达时，不为了动画强制拆成多条。
- 流式输出完成前不再叠加新的“正在思考”气泡。

## 18. 主动提醒、等待与时间

### 18.1 主动提醒的决策输入

- 学生最后一次有意义操作；
- 当前角色阶段的无操作提醒阈值与提醒冷却；
- 学段默认等待时间；
- 学生是否在输入、拍照、画图或移动；
- 位置是否正确；
- 当前情绪和投入信号；
- 已发送提醒数量和冷却时间；
- 教师是否暂停催促。

### 18.2 规则

- 学生处于输入、工具操作或安全移动时不提醒。
- 情绪为烦躁、疲惫或害怕时暂停课程催促。
- 一次只提醒当前小步，不重播完整任务。
- 第一次提醒用关心式询问，第二次可给出一层脚手架。
- 达到课程配置的上限后不再自动提醒，可通知教师。

## 19. 弱网、超时和恢复

### 19.1 性能目标

| 项目 | 目标 |
|---|---:|
| 快捷回复、工具事件、教师指令 | 300ms 内开始响应 |
| 自由文本简单回合 | P50 首批可见内容 ≤ 1.2s |
| 课程知识或多模态回合 | P50 首批可见内容 ≤ 2.0s |
| 首次自动重试 | 最多 1 次 |
| 前端用户可见超时 | 建议 8–12s，按场景调整 |

### 19.2 失败处理

- 模型调用失败不更改 `learningState`。
- 平台只在真实超时或错误后显示连接提示。
- 连接提示不伪装成学习反馈，不自动说学生证据已通过或未通过。
- 学生可以重试本轮、继续离线操作或呼叫教师。
- 恢复后从未完成的当前小步继续。

---

## 20. 重复与信息过载检查

发送回复前，平台检查：

- 本轮是否与上一条智能体消息语义高度相似；
- 是否再次询问已确认信息；
- 是否在同一轮同时要求两个以上学生行动；
- 阶段卡、气泡和工具卡是否重复相同任务说明；
- 是否重复打开仍在等待结果的工具；
- 回复长度是否符合当前学段目标。

当前第一版使用规范化文本指纹拦截近似复读，并结合本轮意图区分社交、情绪、修复和任务流程，避免重复兜底重新催任务。语义相近但措辞差异较大的检测仍需补充向量或结构化动作比较。连续两次确认重复时，当前策略失效，进入 `repair_conversation` 或换用快捷回复。

## 21. 标准对话示例

### 21.1 入场和准备

```text
絮絮：数龙官，你现在看到三大殿三台了吗？
       [已到达] [还在路上] [需要导航]

学生：好了

絮絮：好，位置确认了。你想现在开始，还是先等一下队友？
       [现在开始] [等一下]

学生：开始吧

UI：第1个角色阶段·观其形
    目标：认出螭首，发现它与台基排水的关系

絮絮：先选一处螭首，拍一张能同时看清它和台基的全景。
UI：[打开相机]
```

### 21.2 抱怨和修复

```text
学生：你有毒吧，一直问一样的

絮絮：你说得对，刚才的回应让人很烦。我先停下来听你说；想继续时告诉我“继续”就行。

学生：继续

絮絮：好，已经记下你到位了。现在进入第1阶段。
```

### 21.3 无语义输入

```text
学生：==

絮絮：这个符号我没看懂～你是准备好了，还是想再等一下？
       [准备好了] [再等一下]
```

### 21.4 当前小步求助

```text
学生：我不知道哪个是螭首

絮絮：先找台基边缘一排向外伸的石刻。你看到哪一个嘴部有出水口？
```

### 21.5 宣称完成但证据缺失

### 21.6 课程知识问题

```text
学生：螭首为什么能排水？

来源：课程资料·螭首知识条目
```

## 22. 验收标准

### 22.1 回合理解

- [ ] “好了”、“嗯”、“行”、“可以”、“都好了”能回答当前待回答问题。
- [ ] 学生只确认到达时，下一轮只询问准备状态。
- [ ] 已确认状态不再询问。
- [ ] 同一句同时包含多个明确意图时，能正确更新多个可同时确认的信息。
- [ ] 低置信理解只生成一个澄清问题。

### 22.2 自然对话

- [ ] 寒暄不调用完整课程知识，不强制催任务。
- [ ] 抱怨、疲惫、烦躁和拒绝会暂停流程催促。
- [ ] 输入无语义字符不会进入无限循环。
- [ ] 闲聊或打断后可从原小步恢复。
- [ ] 对话修复会承认具体问题并停止重复。

### 22.3 教学引导

- [ ] 每轮最多一个主要对话动作和一个明确问题。
- [ ] 学生每次只看到当前小步。
- [ ] 课程提问根据当前小步、B2 引导和 B6 脚手架生成。
- [ ] 学段信息量、问题开放度和提示频率符合协议。
- [ ] 反馈指向学生的具体方法、观察或证据。

### 22.4 状态和工具

- [ ] 自由文本无法越过 `tool_result / ai_evaluation / teacher_confirm / location_event` 验收。
- [ ] 工具调用符合当前小步权限，过期结果不更新状态。
- [ ] 安全事件和教师广播可打断任何普通回合。
- [ ] 模型、网络和工具失败保留学生草稿和当前状态。
- [ ] 模型不声称工具已完成、位置已验证或证据已通过。

### 22.5 UI 和消息

- [ ] 普通回合最多一条智能体气泡和一张支持卡片。
- [ ] 角色阶段卡、气泡和工具卡不重复相同文案。
- [ ] 普通流程消息不显示课程知识来源标签。
- [ ] 消息间隔只服务于真实多消息，不通过重复营造对话感。

### 22.6 必测对话集

每门课上线前至少执行：

- 入场肯定、否定、模糊和同时确认用例；
- 寒暄、抱怨、拒绝、疲惫、无语义字符和跑题用例；
- 当前小步求助、知识问题、索要答案和常见误区用例；
- 宣称完成、证据不足、工具失败和重复提交用例；
- 位置拒绝授权、导航失败、离开安全区和教师打断用例；
- 小学低年级、小学高年级、初中和高中各一套端到端角色任务用例。

## 25. 文档冲突的统一结论

1. 所有课程统一使用平台 IP“絮絮”。课程可定义絮絮的本课工作身份、专业语气和阶段行为，不覆盖名称和基础形象。
2. 苏格拉底引导只适用于需要学生发现的教学内容。安全、导航、操作、情绪和允许公开的事实使用直接回答。
3. 脚手架触发阈值由平台按学段提供默认值，课程可在合理范围内覆盖。
4. 课程配置编译为最小必要的运行时上下文，不在每轮对话中注入整门课的全部材料。

## 26. 变更准入

后续修改对话引擎时必须同时满足：

- 状态字段或枚举变化同步更新本协议、公开会话结构和持久化迁移；
- 新增工具或完成方式同步更新课程提交规范、客户端 renderer、服务端 validator 和防泄漏测试；
- 新增自然语言行为先补充契约测试，至少覆盖肯定、否定、模糊、抱怨和重复输入；
- 调整 SSE 事件时同步升级服务端、客户端和第 17 节事件表；
- 课程知识、评估答案和限制字段继续遵守 public/private 编译边界；
- 故宫与四渡赤水两门课都通过编译、对话和工具回归后才能合入。

本协议是 P01 AI 对话引擎、课程编译器、学生端渲染层和教师干预通道的共同运行基线。
